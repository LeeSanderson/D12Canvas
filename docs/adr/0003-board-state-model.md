# Board content is a flat, ID-keyed entity model, not an ownership tree

`Board`'s content — component instances, groups, and (per the connectors/edges ticket) edges — is modeled as three flat, independently-addressable collections (`Components`, `Groups`, `Edges`), each keyed by a GUID assigned at creation. No entity owns another by reference; every relationship — group membership, a future edge's endpoints — is expressed as an ID reference into one of these collections, never nested containment.

This is a direct consequence of the collaboration-readiness non-foreclosure constraint: CRDT/OT frameworks merge state by reconciling independently-addressable entities, and a nested ownership tree would force structural conflict resolution (e.g. two clients concurrently moving an instance into different groups) that a flat, ID-referenced model avoids by construction. GUID identity (not a sequential counter) follows for the same reason — concurrent entity creation needs no coordination.

`ComponentInstance` holds `ComponentTypeKey` (the string key from ADR 0001) plus a boxed `object Props`, `Bounds` (position/size, per ADR 0001), and an explicit `ZIndex` field. `Props` is resolved to its real `TProps` shape via a registry lookup — the registry captures `typeof(TProps)` generically at `RegisterComponent` time, so this is a keyed lookup against a table, never a string-to-`Type` parse. `ZIndex` is a plain field rather than implied by collection order, so re-layering is a single-field write, not a shared-sequence reorder — cheaper to reason about under future concurrent edits.

Entities are mutable classes, mutated in place — matching how `ComponentContainer.razor.cs` already mutates `X`/`Y`/`Width`/`Height` directly during drag/resize. Consequence: the deferred undo/redo history model will need explicit change-tracking (e.g. a command/memento pattern recording what changed and how to invert it), since there's no immutable prior value to snapshot for free.

`Group` is its own entity: `Id` plus `MemberIds`, which may reference component instances or other groups, giving nested groups without a separate tree type. A group's bounds are computed on demand from its members rather than stored, so there's no redundant state to go stale.

`Board` reserves an empty `Edges` collection now, following the same entity pattern; the connectors/edges ticket owns everything about what an edge actually contains.

The viewport-query surface validated in tickets "DOM-rendering performance ceiling for Blazor components" and "Virtualization/windowing mitigation stress test" — a plain O(n) bounding-box scan against a viewport rect plus a tunable overscan margin, no spatial index — is exposed as `Board.GetVisible(viewport, overscan)`, a plain instance method over `Components`. Subset-visibility queries (a group's members, a future edge's anchor) are deferred until a ticket actually needs one; if that happens, extracting the scan into a small stateless core function operating over any bounds-bearing sequence is the intended path, rather than duplicating the loop.

**Considered and rejected:** a nested ownership tree (groups owning child references directly, edges nested inside the components they connect) — rejected for the structural-merge-conflict reason above. Immutable value records replacing entities wholesale per change — would give undo/redo and CRDT-merge semantics almost for free, but rejected for now in favor of matching the existing mutable, imperative drag/resize code; revisit if/when the undo/redo ticket makes the cost of mutable entities concrete.
