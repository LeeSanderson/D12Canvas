# Edges attach via fractional ports on component instances, support floating endpoints, and carry per-edge routing, arrows, and an embedded rich label

`Board.Edges` (reserved empty by ADR 0003) is filled in as another flat, GUID-keyed entity: `Edge { Id, Source, Target, RoutingStyle, SourceArrow, TargetArrow, Label }`. Each endpoint (`Source`/`Target`) is a discriminated `EdgeEndpoint` — a `PortEndpoint { ComponentId, PortId }` referencing a component instance's port by ID, or a `FloatingEndpoint { Point }`, a fixed X/Y that tracks nothing. Floating endpoints exist so a connector dragged from a component and dropped on empty canvas still produces a valid edge, matching standard whiteboard-tool behavior — not every edge is fully attached at both ends.

**Ports** are the attachment surface. Every `ComponentInstance` automatically exposes four standard ports at its border centers — top `(0.5, 0)`, right `(1, 0.5)`, bottom `(0.5, 1)`, left `(0, 0.5)` — expressed as a fraction of `Bounds`, not an absolute offset, so a port's live position is always *derived* from current `Bounds` rather than stored. This is what makes an attached edge track a component through move and resize for free, and what keeps a port valid after resize (an absolute-pixel port could end up outside the shape once `Width`/`Height` change). The end user can add further ports to a specific placed instance at runtime — `ComponentInstance.CustomPorts: List<PortDef>`, `PortDef { Id, FractionX, FractionY }` — the same fractional representation as the four standard ports, just user-authored and instance-scoped rather than type-scoped.

Because ports are either automatic (derived from `Bounds`, needing no per-type declaration) or per-instance runtime state (added by the end user, not the component's developer), **no change to ADR 0001's registration contract is required** — a component type becomes connectable without its author doing anything.

**Routing and direction** are both per-edge choices, not board-wide settings: `RoutingStyle: EdgeRouting { Straight, Orthogonal, Curved }` (default `Straight`), and independently `SourceArrow`/`TargetArrow: ArrowStyle { None, Arrow }` (default `SourceArrow = None`, `TargetArrow = Arrow`) — the common single-directed-arrow case front-loaded as the default, with both ends individually overridable for undirected or bidirectional edges.

**Labels** are rich, not plain strings: an edge's `Label` is a full `ComponentInstance` (reusing ADR 0001's component system — any registered type can serve as a label, defaulting to a built-in text component fleshed out by the built-in shape set ticket). Unlike board content, a label is *embedded* directly on the `Edge`, not a separate entry in `Board.Components` — it has no independent identity on the board, is never selected or moved on its own, and is deleted automatically with its owning edge. This is a deliberate exception to ADR 0003's flat-entity pattern: a label's lifecycle is fully owned by its edge, so giving it independent board-entity status would only introduce an orphan-management problem with no offsetting benefit.

**Scope boundary:** an edge attaches to component instances (or floats) — attaching an edge to a whole `Group` as a single unit is not decided here. It's carried forward as a "not yet specified" item, since it depends on how the still-open selection/grouping model treats a group as an attachable thing.

**Considered and rejected:**
- **Dynamically-calculated, unnamed anchor points** (always route to the nearest point on the target's bounding-box perimeter, no discrete ports) — rejected; the user wants a concrete, named, addressable set of connection points (matching precise-diagramming tools), not an implicit nearest-point calculation.
- **Ports declared at registration time only** (component developer opts a type into specific anchors) — rejected in favor of automatic standard ports (no opt-in needed) plus end-user-added custom ports; keeps ADR 0001 untouched and puts port authorship where the user actually wants it — in the end user's hands at runtime, not the component developer's at registration.
- **Absolute pixel offsets for custom ports** — rejected; a port placed at, say, `(180, 0)` on a 200px-wide instance ends up outside the shape if resized to 100px wide. Fractional offsets are correct by construction across any resize.
- **A single board-wide routing style / arrow setting** — rejected in favor of per-edge choice, consistent with how every other per-instance visual property in this model (bounds, z-index) is already per-entity, not board-global.
- **Label as a full `Board.Components` entity**, ID-referenced from the edge (mirroring `Group.MemberIds`) — rejected in favor of an embedded value; a label has no independent existence without its edge, so first-class entity status would only add an orphan-cleanup concern the embedded form avoids entirely.
