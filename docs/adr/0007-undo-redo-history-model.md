# Undo/redo is a gesture-level command stack over generic entity mutations, capped and session-scoped

Undo/redo tracks history at the granularity of a completed user gesture — a drag, a resize, a prop edit, a create, a delete, a group, an ungroup — recorded once on gesture commit (pointer-up/blur), never per intermediate frame. This follows directly from `Board`'s entities being mutable classes mutated in place (ADR 0003): there's no immutable prior value to snapshot for free, so each gesture must record an explicit, invertible change. Selection is excluded entirely — ADR/ticket "Selection model" already ruled selection out of undo/redo, since it's transient view state with no meaning to a reload or a second collaborator.

The command set is a small, closed group of primitives rather than one bespoke class per user-facing action, mirroring `Board`'s own flat, generic entity model instead of inventing a parallel taxonomy of gesture types:

- `AddEntity(entity)` / `RemoveEntity(id)` handle create/delete for any entity — component instance, group, or edge. The entity object already carries its full state at the point of removal, so no per-type command is needed to reconstruct it on undo.
- `ChangeBoundsCommand(before: Bounds, after: Bounds)` covers both move and resize as one command, since `Bounds` is already a single combined position+size struct (ADR 0001/0003) and a corner-drag gesture can change both simultaneously. Move vs. resize is inferred after the fact from which fields stayed fixed (width/height unchanged ⇒ move; top-left unchanged ⇒ resize), not tracked as two separate command types.
- `MutateEntity(id, beforeProps, afterProps)` is a generic swap for a component instance's opaque, boxed `Props` (ADR 0001/0003). Because `Props` is only ever resolved via a registry lookup, no per-component-type undo logic is possible or needed — this matches the registration contract's promise that a host app supplies `Props`, never undo behavior.
- `GroupCommand` / `UngroupCommand` are named for readability in a history/log context, but are thin wrappers over `AddEntity`/`RemoveEntity` acting on the `Group` entity itself. Because `Group.MemberIds` is a reference list held by the group (not a back-pointer on members), grouping/ungrouping never mutates the member instances — the wrapper only needs to add or remove the `Group` entity.
- `CompositeCommand` wraps an ordered list of child commands into one atomic history-stack entry, applied/inverted together (children inverted in reverse order on undo). This is required whenever a single user gesture produces more than one primitive/named command — e.g. a multi-select move or resize (one `ChangeBoundsCommand` per selected instance), a future delete-selection (one `RemoveEntity` per instance), or paste (one `AddEntity` per pasted instance).

History is capped at 1000 entries, implemented as a circular buffer, and lives purely in memory for the current session — it is not part of the persisted envelope (the `IBoardSerializer` format from ADR 0004 has no history field), so reloading a board always starts with empty undo/redo stacks.

No actor/origin field is attached to commands. Undo is local-user-only by design: it only ever reverts the local user's own changes. This is a deliberate non-decision, not an oversight — if/when real-time multi-user collaboration becomes its own future effort, keeping each user's undo scoped to their own changes is that effort's problem to solve, likely needing its own attribution and per-user stack mechanism. Nothing here stubs, reserves, or anticipates that shape.

**Considered and rejected:**
- **Semantic per-action command classes** (`MoveCommand`, `ResizeCommand`, `CreateComponentCommand`, `DeleteGroupCommand`, etc., one per operation) — rejected in favor of the small generic primitive set; `Board`'s own state model is already flat and generic, and per-type commands would duplicate the same apply/invert logic a generic swap already covers, especially for opaque `Props` where no type-specific behavior is even possible.
- **Reserving an unused `OriginId` field now**, following the `SchemaVersion`-style "reserve now, use later" pattern (ADR 0004) — rejected; undo is explicitly local-only, so there is no near-term consumer for the field, and adding it risks inviting speculative per-user filtering logic before a second user exists.
- **Unlimited history depth** — rejected in favor of a bounded circular buffer (1000 entries), so memory use across a long editing session has an explicit ceiling rather than growing indefinitely.
